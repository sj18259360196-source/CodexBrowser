# 2026-07-13 - Initial MVP

## Objective

Build the first usable Codex-specific research browser: a visible desktop Chromium session that preserves authentication, alerts the user when human authorization is required, downloads and reads PDFs, and exposes the workflow to Codex through a local plugin and MCP server.

## Decisions

- Use Electron with a persistent Chromium partition instead of building a browser engine.
- Keep the desktop process alive in the system tray so session cookies can remain active.
- Use a local Windows named pipe (`codex-browser-v1`) between the MCP process and the desktop process.
- Keep passwords, cookies, authentication headers, and raw profile files outside the MCP surface.
- Put durable browser/session/download/PDF capabilities in the core and keep future university or publisher rules as adapters.
- Return PDF text by page instead of sending an entire document in one MCP result.

## Known Issues

- The current terminal cannot run the WindowsApps `codex.exe`, so `codex plugin add codex-browser@personal` is blocked by an access-denied error. The marketplace entry and validated plugin source are ready for installation through the Codex app.
- Only the primary browser profile exists; institution profile management is not implemented yet.
- No university-specific or publisher-specific adapters have been added because the actual institution and target databases have not been supplied.
- Scanned PDF OCR, SQLite/FTS5, multiple tabs, and packaged Windows installers remain future work.
